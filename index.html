Adobe Captivate and Articulate Storyline

Posted Date: Mar 10, 2016

Adobe Captivate and Articulate Storyline are used for html and elearning projects. I worked through the main features of both programs and created the demos listed below. It was fun to try both products and I would enjoy working more with either, if a project came up.

Demo 1 - Computer Security
Articulate Storyline project (non-responsive, scales to fit screen). This demo uses a few of Storyline’s quiz features.

Demo 2 - Slider
Articulate Storyline project (non-responsive, scales to fit screen). This slider uses transitions and layers, as well as some of the animation features available in Storyline.

Demo 3 - Garmin
Adobe Captivate project (responsive). I got this idea on a bike ride when a friend told me she didn’t know what the buttons on her Garmin do. She was afraid to press the wrong button. She always rides with her phone on hand, so the idea would be to have a touch based help description of the basic features to consult over coffee, or on the go.

Demo 4 - Survey
Adobe Captivate project (responsive). This survey uses Captivate variables to track the answers to the questions and then displays the survey results in the final screen. It’s a bit long for a demo, but if you click through you’ll come to the results screen at the end.

Demo 5 - Colorbars
Adobe Captivate project (non-responsive). I got the idea from a challenge on the Articulate Storyline site and decided to create it in Captivate while testing different Captivate features.

Do I need Sass or Less?

Posted Date: Oct 16, 2015

In reading around on the web about how to implement Wintersmith, I came upon many blogs about the benefits of using Sass/Less for writing and maintaining CSS.

Sass, Less, Coffeescript, and HAML are all preprocessors. Wintersmith uses the Coffeescript preprocessor and the Jade template engine, which is based on the HAML preprocessor.

Preprocessors are designed to make it easier to maintain code. The code is written in the preprocessor format, and then is converted to CSS/Javascript/HTML so that the information can be read on the web.

  • Sass and Less translate into CSS
  • Coffeescript translates into Javascript
  • HAML translates into HTML

There’s a lot of information there. I’ll definitely be leaving Coffeescript and HAML/Jade until a later date. But would it make sense to start now with Sass or Less now?

Wintersmith - Inserting "Read More" breaks

Posted Date: Oct 8, 2015

Wintersmith can be set to display the first section of a text with a ‘read more’ link to the full text.

I learned from Brian Rinaldi’s site that the “read more” link is automatically inserted whenever one of the following is inserted into an article page:

  • a heading at the <h2> level
  • the <hr> tag
  • the span code: <span class="more"></span>

I’ve started using the <hr> tag and it seems to be working. I updated the css file for <hr> with “display:none” so that a line doesn’t display in the post.

Posted Date: Oct 7, 2015

When using with different HAT systems, I tend to work with two or three image styles. In basic Markdown, there is only the img tag itself to work with. However, classes can be added by inserting html into the Markdown:

Basic Markdown: ![Image Ideas](/img/manageimages.png) versus
HTML entered into Markdown: <img class="testing" src="/img/manageimages.png">

Both methods just use one line, so it does not add complexity to use the HTML syntax. For now I’ll just use the basic Markdown syntax with an img (centered, with a max-width of 100%). When I need more styling, I’ll start using the HTML syntax.

Why Wintersmith?

Posted Date: Oct 6, 2015

I’ve worked with Help and Manual, RoboHelp, and MadCap Flare and I’ve become accustomed to being able to single source to webhelp, pdf, chm and other formats quickly.

Each of the commercial help authoring products has great features, and if someone was to give me a free copy, I would happily work away. However, Murphy’s Law states that if I buy one of these products, my next project will be in one of the others. And by the time I return to the product I purchased it will be out of date. For writing systems options abound. In all sorts of directions. Everything seems so expensive. Lower cost options: OxygenXML? Jekyll and git?

OxygenXML has a $200 version for home use. I downloaded the trial version and created a simple DITA project and then wanted to test the outputs. The company documentation includes a topic about PDF customization: DITA to PDF Customization, but I could not get it to work. I emailed customer support to ask if a person would need to know XSLT to change the default PDF output and the response was yes, it requires a knowledge of XSLT OR it can be done using CSS customizations and the commercial Prince XML engine (cost $500US). The cost of buying both OxygenXML and Prince brings the purchase of OxygenXML up to about the same cost as the help authoring software. So that ended the OxygenXML experiment.

I looked at Jekyll but read that it works best on Mac, and although I have a Mac, I do most of my work on Windows.
